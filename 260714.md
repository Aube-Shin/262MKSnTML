## 개요: 개발 진행 개요- 아트 진행 개요
### 개발 진행 개요
- A/B파트로 나누어서 진행, 오류 발생을 최소화하기 위해 로직 관리와 씬 관리로 분리.
- 로직 관리 파트를 A파트로 명명, 시각적인 요소나 씬 관리 없이 전체 로직 관리만 담당.
- 씬 관리 파트를 B파트로 명명, 2.5D 구현을 포함한 씬 내의 모든 시각적 요소의 관리를 담당.
- A파트 호연이가 B파트 준우가 맡기로 정함.

### 아트 진행 개요
- 아트 역시 A/B로 파트를 나누되, 개발과 달리 작업 분담에 가까운 형태로 진행.
- A파트는 실외 요소(건물, 가로수, 등 실외 맵 구현에 필요한 요소)와 UI를 담당.
- B파트는 실내 요소(가구, 소품 등 실내 맵 구현에 필요한 요소)와 캐릭터(플레이어, NPC)를 담당.
  - 처음 계획했을 땐 실외 요소가 작업량이 많아 이렇게 정했으나, 추후 조정 가능성 존재함.
- A파트는 이현이가, B파트는 내가 담당하고 추가적으로 작업량이 적은 내가 노말맵 변환까지 맡기로 정함.

## 개발 A 파트: 전체 구상-TimeManager.cs-ClueSO.cs
### 전체 구상
- 개발 A파트는 사전에 내가 미리 만든 프로토타입을 바탕으로 설명함.
  - 다만 실질적인 제작은 내가 아니라 호연이가 하므로 내가 만든 프로토타입을 뜯어보면서 작업하라고 전달함.
- A파트는 맵이 있는 씬에 손대지 않기로 합의했으므로 'Bootstrap'이라는 씬 1개만을 사용함.
  - 게임은 항상 이 씬에서 시작하며, Bootstrap.cs라는 스크립트가 시작과 동시에 Additive로 실제 시작 씬을 불러오게 만듬.
- bootstrap.cs가 붙은 'CoreSystem'이라는 빈 오브젝트에 각종 매니저들을 자식으로 단 구조를 사용함.
  - 내 생각엔 부모 오브젝트인 bootstrap.cs에 DontDestroyOnLoad를 붙이면 자식 객체들도 사라지지 않을 것 같은데.. 불안해서 테스트를 안해봄 추후 다시 확인 필요.
- 프로토타입 내 모든 매니저 스크립트를 설명하기엔 어렵기도 하고 굳이 필요도 없어서 필수적이라고 생각하는 두 스크립트만 소개하고 넘어가기로 결정함.

### TimeManager.cs
- 시간대가 중요한 역할을 하는 게임인만큼 이를 관리하는 TimeManager.cs는 다른 매니저 스크립트도 참조하는 우선순위가 가장 높은 스크립트임.
  - 다른 매니저에서 참조하는 경우가 많으므로 옵저버 패턴을 이용함.
  - 열거체 방식으로 시간대를 관리함.
  - 특정 사건(예: 하루의 끝)을 event Action으로 관리해 참조가 용이하게 만드려고 함.
  - 현실의 n초가 인게임 1시간이 되는 구조로 시간을 흐르게 만들었으며, 추후 어느정도가 적당한지 정해야 함.

### ClueSO.cs
- Scriptable Object를 생성하여 단서를 생성할 예정이고, 이를 관리하는 스크립트는 ClueManager.cs임.
  - 여기서 A파트는 ClusSO.cs만 관리하며, 실제 SO를 사용하는 건 B파트가 하게 될 거임.

## 개발 B 파트: 전체 구상- 2.5D- 노말맵- 독립적인 씬 목록- 구성 양식
### 전체 구상
- B파트의 중요 부분 중 하나는 2.5D의 구현인데, 이를 3D 오브젝트를 이용할지 아님 2D 오브젝트만 가지고 해볼지 몇몇 프로토타입을 만들어서 테스트한 뒤 B파트를 맡은 준우랑 협의함.
- 결론적으로 노말맵을 사용한 방식으로 구현하기로 하였으며, 3D 오브젝트는 사용하지 않는 것으로 결정됨.
  - 게임 엔진은 Unity 3D(URP)를 활용하며, URP Material에 노말맵을 넣어 양감을 더하기로 정함.
  - 카메라는 Cinemachine을 활용해 횡스크롤 시점을 표현하려고 계획중임.

### 2.5D
- 2.5D 환경의 경우, 크게 플레이어/건물, 배경, 바닥이라는 3요소로 나누어서 논의를 진행함.
- 먼저 플레이어/건물의 경우, 큰 이변이 없는 한 서로 z축을 다르게 하여 거리를 표현할 것 같음.
- 배경의 경우, 일단 난 프로토타입에서 z축에 큰 차이를 두어 2D 오브젝트를 뒤에 띄우는 방식으로 진행함.
  - 하지만 추후에 skybox를 사용할 가능성도 있다고 생각됨.
  - 나는 프로토타입에서 skybox를 None으로 설정 후 전역 조명과 Bloom 기능을 활용해 표현했음.
- 마지막으로 바닥의 경우, 내 프로토타입엔 Terrain이 사용되었으나, 테스트해보면 그냥 3D Cube를 길게 늘려 사용해도 문제는 없음.
  - 다만 Terrain을 사용할 시 텍스쳐 등을 좀 더 사실적으로 구현할 수 있을 것 같아 Terrain으로 프로토타입을 제작함.

### 노말맵
- 2.5D 구현은 노말맵을 활용하기로 함.
- B파트 담당자는 이미 노말맵을 잘 알고 있었지만 다른 파트도 일단 알아야 할 것 같아서 간단히 설명함.
- URP/Complext Lit으로 각종 요소의 Material을 제작해 노말맵을 적용시키기로 결정함.
  - 노말맵 스프라이트의 경우 꼭 'Normal map'으로 타입을 변경해야 함.
  - 구체적인 변환 계획은 뒤에 아트 파트에서 설명하기로 하고 넘어감.

### 독립적인 씬 목록
- 하나의 독립적인 씬(맵)을 사용할 부분을 목록으로 정리함
- 실외 맵은 '제페론 도심'과 '포츠벨트 외곽'으로 총 2개임
- 실내 맵은 약 30개로, 건물 당 필요한 실내 맵의 개수가 서로 달라 이렇게 많은 요구량이 산출됨.
  - 실내 맵의 경우 추후 약간의 조정 사항이 있을 것으로 생각되지만, 그래도 개수가 25개 이하로 줄어들진 않을 것 같음.
  - 예상했던 것보다 훨씬 작업량이 많은 것 같아 추후에 내가 방해가 안 되는 선에서 도와주면 좋을 것 같음.

### 구성 양식
- 마지막으로, B파트가 시각적인 구현 요소를 담당하는 만큼 각종 씬의 건물 배치도를 어떤 형태로 전달해야 할 지 논의함.
  - 실외/실내 맵 구성표를 하나 예시로 만들었는데, 이 정도면 괜찮을 것 같다고 하여 이대로 진행할 예정임.
- 그리고 상호작용 가능한 물건들의 상호작용 내용을 어떻게 전달해야 할 지 논의함.
  - 마찬가지로 표 형태의 예시를 하나 만들었는데, 괜찮다고 해서 기본적인 내용은 표의 형태로 작성하고 세부 내용은 docs 등의 문서로 전달하기로 결정함.

## 아트 파트: 작업 방향 및 시안 소개- 노말맵 변환

  - 개발 파트에게 스케치 시안을 보여줌.
  - 사전에 협의한 내용과 큰 차이 없이 작업 진행할 것으로 예상됨.(스케치 시안 수정도 X)

### 노말맵 변환
- 노말맵은 원래 직접 색을 입히는 게 맞지만, 그보다 편리한 변환 사이트를 발견해 이를 이용할 예정임.
  - 기본적으로 모든 변환은 'NORMALMAP ONLINE'이라는 온라인 사이트를 이용함.
  - 만약 디테일한 조정이 필요할 경우, 'Laigter'라는 프로그램을 추가적으로 이용할 계획임.
